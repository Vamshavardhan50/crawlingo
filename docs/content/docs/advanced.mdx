### Execution Engine

The execution engine follows this flow for every fetch request:

```
1. URL Parsing (url::Url)
2. Host extraction for rate limiting
3. Rate limiter wait (governor token bucket)
4. Strategy selection (Standard vs Stealthy)
5. HTTP request via wreq client
6. Response body collection
7. HTML parsing (lol-html streaming)
8. DomTree construction (flat Vec<DomNode>)
```

### Fetcher Engine

Two strategies implement `FetchStrategy` trait:

- `HttpFetcher`: Standard HTTP/HTTPS client with connection pooling
- `BrowserFetcher`: TLS-fingerprinted stealth fetcher (currently delegates to HttpFetcher for Phase 3)

Strategy selection is based on `FetcherTier`:

```rust
pub enum FetcherTier {
    Standard,
    Stealthy,
}
```

### Session Management

The `Session` struct stores thread-safe configuration:

```rust
pub struct Session {
    headers: RwLock<HashMap<String, String>>,
    cookies: RwLock<HashMap<String, String>>,
    proxy: RwLock<Option<String>>,
    rate_limit_rps: RwLock<f64>,
    auto_match: RwLock<bool>,
    timeout_seconds: RwLock<u64>,
    fingerprint_path: RwLock<String>,
    fetcher_tier: RwLock<FetcherTier>,
    browser_profile: RwLock<Option<String>>,
    similarity_weights: RwLock<HashMap<String, f64>>,
    proxy_pool: RwLock<Vec<String>>,
    proxy_index: Arc<AtomicUsize>,
    proxy_provider_url: RwLock<Option<String>>,
    fingerprint_store: RwLock<Option<(String, Arc<FingerprintStore>)>>,
}
```

All fields are `RwLock`-protected inside an `Arc<Session>`, enabling shared concurrent access.

### Caching

| Cache | Location | Lifetime | Purpose |
|-------|----------|----------|---------|
| CSS Selectors | Global `DashMap<String, CompiledSelector>` | Process lifetime | Avoid re-parsing CSS queries |
| Regex Patterns | Global `DashMap<String, Regex>` | Process lifetime | Avoid re-compiling regex |
| DNS | `moka::Cache<String, IpAddr>` | Configurable TTL (default 60s) | Avoid repeated DNS lookups |
| Fingerprints | Sled database on disk | Persistent across runs | Store DOM fingerprints for auto-match |

### Memory Management

- **DOM Tree:** Flat `Vec<DomNode>` with `usize` index references вҖ” no pointers, no reference counting
- **Page struct:** Stores `Arc<DomTree>` вҖ” cheap cloning for FFI sharing
- **Text extraction:** Lazily computed via `OnceCell<String>`
- **Element objects:** Lazily created from index on access

### Concurrency Model

- **Tokio** async runtime for all I/O (fetching, rate limiting)
- **Rayon** parallel iterators for CPU-bound work (auto-match scoring, change detection)
- **JoinSet** for concurrent crawl workers
- **Mutex** for shared state (visited set, pending queue)
- **DashMap** for lock-free concurrent hash maps (rate limiters, caches)

### Scaling

For large-scale crawling:

1. **Per-host rate limiting** via `HostRateLimiter` (token bucket per host)
2. **Connection pooling** via wreq (max 100 idle connections, 90s idle timeout)
3. **Concurrent workers** via `Crawl::concurrency(n)` (JoinSet of n tasks)
4. **Scheduling** via `Crawl::schedule(interval)` (dedicated thread with infinite loop)
5. **Webhooks** for real-time streaming of results

### Observability

- Internal `tracing` crate for structured logging
- WARN-level logs for auto-match recovery events
- ERROR-level logs for crawl failures
- INFO-level logs for scheduled crawl execution

### Security

- Proxy authentication supported (user:pass in proxy URL)
- Cookie jar isolation per Session
- TLS fingerprint emulation for stealth fetcher
- Rate limiting prevents accidental DoS

### Ethical Scraping

Always follow these guidelines:

1. **Respect robots.txt** вҖ” Crawlingo does not automatically enforce robots.txt; you must configure `follow` selectors responsibly
2. **Rate limit** вҖ” Always set `session.rate_limit(n)` to avoid overwhelming servers
3. **Set a descriptive User-Agent** вҖ” Identify your bot so site owners can contact you
4. **Cache when possible** вҖ” Avoid repeated fetches of the same URL
5. **Respect copyright** вҖ” Only scrape data you have permission to use
